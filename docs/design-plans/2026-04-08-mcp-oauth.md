# MCP OAuth Authentication Design

## Summary

UltraBridge's MCP server (`ub-mcp`) exposes note search and retrieval tools to AI agents over HTTP SSE. Currently it supports Basic Auth and a static bearer token set by an environment variable. This design replaces that static token with a database-backed bearer token system, enabling persistent, revocable tokens that can be issued per-client — the immediate use case being claude.ai's remote MCP connector, which expects a fixed `authorization_token` in its config.

The approach avoids any external identity infrastructure. A new `internal/mcpauth/` package implements token CRUD directly against the shared SQLite notes database (`notedb`). Tokens are generated from 32 bytes of cryptographic randomness, and only the SHA-256 hash is stored — identical to the GitHub personal access token model. `ub-mcp` validates incoming bearer tokens per-request via a direct SQLite query; the ultrabridge web UI gains a settings card for creating and revoking tokens. The existing static env var and Basic Auth paths continue to work, keeping backward compatibility.

## Definition of Done

1. **ub-mcp accepts bearer tokens** on its HTTP SSE transport, validated against a SQLite-backed token table. Basic Auth continues to work.
2. **ultrabridge web UI** has a page for creating and revoking MCP bearer tokens (username + generated token).
3. **claude.ai can connect** to ub-mcp by configuring the issued bearer token as `authorization_token`.
4. **No external dependencies** — no Keycloak, no external OAuth provider. Self-contained.

## Acceptance Criteria

### mcp-oauth.AC1: Token store operations
- **mcp-oauth.AC1.1 Success:** `CreateToken` generates a 43-char URL-safe base64 token and stores its SHA-256 hash
- **mcp-oauth.AC1.2 Success:** `ValidateToken` returns the label for a valid token
- **mcp-oauth.AC1.3 Success:** `ValidateToken` updates `last_used` timestamp on successful validation
- **mcp-oauth.AC1.4 Failure:** `ValidateToken` returns error for a token not in the database
- **mcp-oauth.AC1.5 Failure:** `ValidateToken` returns error for a revoked (deleted) token
- **mcp-oauth.AC1.6 Success:** `ListTokens` returns all tokens with label, truncated hash, created_at, and last_used

### mcp-oauth.AC2: ub-mcp bearer token authentication
- **mcp-oauth.AC2.1 Success:** HTTP SSE request with valid DB-backed bearer token is accepted
- **mcp-oauth.AC2.2 Success:** HTTP SSE request with valid static `UB_MCP_AUTH_TOKEN` is accepted (backward compat)
- **mcp-oauth.AC2.3 Success:** HTTP SSE request with valid Basic Auth credentials is accepted
- **mcp-oauth.AC2.4 Failure:** HTTP SSE request with invalid/missing bearer token returns 401
- **mcp-oauth.AC2.5 Edge:** If neither `UB_DB_PATH` nor `UB_MCP_AUTH_TOKEN` is set, startup logs a warning

### mcp-oauth.AC3: Web UI token management
- **mcp-oauth.AC3.1 Success:** Creating a token from settings page displays the raw token once
- **mcp-oauth.AC3.2 Success:** Token list shows label, truncated hash (8 chars), created date, last used date
- **mcp-oauth.AC3.3 Success:** Revoking a token removes it from the list and immediately invalidates it
- **mcp-oauth.AC3.4 Failure:** Creating a token with empty label is rejected
- **mcp-oauth.AC3.5 Edge:** Raw token is not visible after navigating away from the flash page
- **mcp-oauth.AC3.6 Success:** Revoked token returns 401 on next ub-mcp request (cross-component verification)

### mcp-oauth.AC4: claude.ai connectivity
- **mcp-oauth.AC4.1 Success:** A token created via the web UI works as `authorization_token` in claude.ai MCP connector config

## Glossary

- **MCP (Model Context Protocol)**: An open protocol for exposing tools and data to AI agents. `ub-mcp` is UltraBridge's MCP server; it lets clients like claude.ai call tools such as `search_notes` and `get_note_image`.
- **HTTP SSE (Server-Sent Events)**: A transport that ub-mcp uses for its HTTP mode — the server pushes a stream of events to the client over a persistent HTTP connection. MCP uses SSE to send tool responses.
- **Bearer token**: An opaque credential sent in the `Authorization: Bearer <token>` HTTP header. The server validates the token and grants access if it is recognized.
- **notedb**: The shared SQLite database used by UltraBridge's notes pipeline, Boox pipeline, and chat subsystem. This design adds an `mcp_tokens` table to it, making it the token store for ub-mcp as well.
- **SHA-256 hash**: A one-way cryptographic digest. Storing the hash instead of the raw token means a database breach does not expose usable credentials.
- **URL-safe base64**: A variant of base64 encoding that replaces `+` and `/` with `-` and `_`, making the result safe to include in URLs and HTTP headers without percent-encoding.
- **WAL mode (Write-Ahead Logging)**: A SQLite journal mode that allows concurrent readers while a single writer is active. UltraBridge uses it so both the main process and ub-mcp can open the same SQLite file safely.
- **Flash pattern (query param flash)**: A UI technique where a one-time value (here, the raw token) is passed to the next page as a URL query parameter and displayed once. No server-side session is required.
- **Idempotent migration**: A schema migration written as `CREATE TABLE IF NOT EXISTS`, so it can be run on every startup without error, regardless of whether the table already exists. The pattern used throughout UltraBridge's SQLite schema management.

## Architecture

Direct DB validation: ub-mcp opens the shared notedb SQLite database (via `UB_DB_PATH`) and validates bearer tokens per-request against an `mcp_tokens` table. No in-memory cache — SQLite query latency is negligible at MCP request rates, and skipping the cache keeps token revocation instant.

Token security follows the GitHub personal access token model: raw tokens are generated from 32 bytes of `crypto/rand` (URL-safe base64, 43 chars), and only the SHA-256 hash is stored. The raw token is displayed once at creation and never retrievable again.

**Components:**

- **`internal/mcpauth/`** — New package. Token CRUD and validation as stateless functions operating on `*sql.DB`. Both ub-mcp and ultrabridge import this package.
  - `CreateToken(ctx, db, label) → (rawToken, tokenHash, error)`
  - `ValidateToken(ctx, db, rawToken) → (label, error)` — hashes input, queries table, updates `last_used`
  - `ListTokens(ctx, db) → ([]TokenInfo, error)`
  - `RevokeToken(ctx, db, tokenHash) → error`
  - `Migrate(ctx, db) → error` — idempotent `CREATE TABLE IF NOT EXISTS`

- **`cmd/ub-mcp/main.go`** — Opens notedb on startup via `UB_DB_PATH`. Replaces existing `bearerAuth` function with DB-backed middleware using `mcpauth.ValidateToken`. Auth chain on HTTP SSE: DB token → static `UB_MCP_AUTH_TOKEN` env var (deprecated) → Basic Auth.

- **`internal/web/`** — Settings page extended with MCP Tokens card. Token table (label, truncated hash, created, last used, revoke button) plus create form. One-time token display via query param flash pattern.

**Schema (`mcp_tokens` table in notedb):**

```sql
CREATE TABLE IF NOT EXISTS mcp_tokens (
    token_hash TEXT PRIMARY KEY,
    label      TEXT NOT NULL,
    created_at INTEGER NOT NULL,
    last_used  INTEGER NOT NULL DEFAULT 0
)
```

- `token_hash`: SHA-256 hex of raw bearer token
- `label`: human-readable identifier (e.g. "claude.ai")
- `created_at`: millisecond UTC unix timestamp
- `last_used`: updated on each successful validation; 0 = never used

**Auth middleware chain (ub-mcp HTTP SSE):**

1. Extract `Authorization` header
2. If `Bearer <token>`: SHA-256 hash, query `mcp_tokens`. On match → update `last_used`, allow. On miss → try step 3.
3. If `UB_MCP_AUTH_TOKEN` is set and matches → allow (deprecated path).
4. If `Basic` auth: validate against `UB_MCP_API_USER`/`UB_MCP_API_PASS` → allow or reject.
5. No valid auth → 401 Unauthorized.

**Token lifecycle:**

- **Create:** ultrabridge web UI → `mcpauth.CreateToken` → store hash in notedb → display raw token once
- **Use:** claude.ai sends `Authorization: Bearer <raw_token>` → ub-mcp validates via `mcpauth.ValidateToken`
- **Revoke:** ultrabridge web UI → `mcpauth.RevokeToken` → DELETE row → next request with that token gets 401

## Existing Patterns

This design follows patterns established in the codebase:

- **Database functions:** `internal/mcpauth/` follows the `internal/notedb/` settings pattern — stateless functions taking `(ctx, *sql.DB, ...)` rather than a struct with state. See `notedb.GetSetting`/`notedb.SetSetting`.
- **Auth middleware:** The ub-mcp `bearerAuth` wrapper is a simple `http.Handler` middleware. The new DB-backed version follows the same shape. `internal/auth.Middleware` (Basic Auth) uses a `New()`/`Wrap()` pattern, but ub-mcp's middleware is simpler since it doesn't need a struct.
- **Schema migration:** Uses idempotent `CREATE TABLE IF NOT EXISTS`, matching `internal/notedb/schema.go` and `internal/taskdb/schema.go`.
- **Settings UI:** The MCP Tokens card follows the existing settings page layout — cards with forms that POST to specific endpoints and redirect on success. See `handleSettings`/`handleSettingsSave` in `internal/web/handler.go`.
- **Config:** ub-mcp reads env vars directly in `main.go` (not via `internal/config/`). `UB_DB_PATH` is the only new env var dependency for ub-mcp.

**New pattern:** One-time flash display via query parameter (`/settings?new_token=...`). No existing flash mechanism exists. This is a pragmatic choice for a single-user system — avoids adding session infrastructure.

## Additional Considerations

**Shared SQLite access:** Both ultrabridge and ub-mcp open the same notedb file. SQLite WAL mode with `MaxOpenConns=1` on each side handles this safely — WAL allows concurrent readers with a single writer. Token writes are infrequent (create/revoke from UI only), so write contention is not a concern.

**`UB_MCP_AUTH_TOKEN` deprecation:** The static env var continues to work for backward compatibility. No removal timeline. If both DB tokens and the env var are configured, DB is checked first. This allows gradual migration.

**Token in URL flash:** The raw token briefly appears in the URL bar after creation (`/settings?new_token=...`). For a single-user home system behind Basic Auth, this is acceptable. The token disappears on next navigation. Browser history may retain it — acceptable trade-off for avoiding session infrastructure.
